# RMDB 项目 Agent 工作约束

## 项目定位

本仓库是 2023 数据库系统设计赛提供的 RMDB 关系型数据库管理系统框架，主要用于教学、竞赛和数据库内核能力训练。

不要把当前代码误判为已经完整实现的数据库产品。仓库中大量模块保留了 `Todo` / `TODO` 框架函数，例如磁盘管理、缓冲池、记录管理、B+ 树索引、执行器、事务、锁和恢复等。处理问题前应先确认目标模块的真实完成度。

## 代码结构认知

核心代码位于 `rmdb/src`：

- `storage`：磁盘文件、页、缓冲池管理。
- `replacer`：缓冲池替换策略。
- `record`：记录文件、位图、表扫描。
- `index`：B+ 树索引。
- `system`：数据库、表、索引元数据管理。
- `parser`：SQL 词法语法解析和 AST。
- `analyze`：语义分析、列名解析、条件检查。
- `optimizer`：查询计划生成和简单优化框架。
- `execution`：SQL 执行器和执行管理。
- `transaction`：事务对象、事务管理和锁管理。
- `recovery`：日志和崩溃恢复。

服务端入口是 `rmdb/src/rmdb.cpp`，客户端入口是 `rmdb/rmdb_client/main.cpp`。单元测试入口主要是 `rmdb/src/unit_test.cpp`。

## 工作原则

- 修改前先阅读相关模块的头文件、实现文件、测试和调用链，避免只补局部函数却破坏上层约定。
- 搜索代码和文件时不要使用 `rg`，改用 `grep`、`find` 等工具。
- 优先沿用现有数据结构、错误类型、命名风格和 C++17 写法。
- 保持改动聚焦，不做无关重构、格式化或目录整理。
- 遇到已有未提交改动，默认视为用户改动，不得回滚、覆盖或顺手清理。
- 实现一个功能时，同步考虑它与上下游模块的接口契约，例如页 pin/unpin、dirty 标记、元数据落盘、索引维护、事务日志和异常路径。
- 对比赛框架中明确留给参赛者实现的 TODO，按当前需求补完必要范围，不擅自扩大到完整工业级数据库设计。
- `DATETIME` 固定使用 `YYYY-MM-DD HH:MM:SS` 格式和 8 字节 `YYYYMMDDHHMMSS` 整数编码；输入校验、编码与格式化应复用 `common/datetime.h`，不要在各执行器重复实现。
- 聚合查询当前仅支持无 `GROUP BY` 的纯聚合列表：`COUNT/MAX/MIN/SUM` 可带 `AS` 别名和 `WHERE`，不得与普通投影列或 `ORDER BY` 混用。
- 块嵌套循环连接默认使用 64MB join buffer，并惰性分配、复用固定连续空间，按真实字节限制上限，避免容器扩容产生额外内存峰值；优先缓存过滤后较小的右侧记录块，候选记录对直接在左右原始记录上判断条件，只为匹配结果构造输出，避免逐候选分配临时 `RmRecord`。
- 查询优化必须面向通用关系和谓词规则，不得针对特定 SQL、表名、列名或常量硬编码；等值内连接可选择 Hash Join，固定使用不超过 64MB 的连续哈希空间，超限时安全回退块嵌套循环连接，所有原连接条件仍需在候选匹配后复核。
- 当前锁模型只有表锁和记录锁；可串行化扫描（顺序扫描和索引扫描）使用表级 `S` 锁覆盖查询谓词并持有到事务结束，以阻止并发 `INSERT` 产生幻读。写操作继续使用表级 `IX` 与记录级 `X` 锁，更新或删除事务中的 `S + IX` 按 `SIX` 处理。
- 故障恢复采用无检查点 WAL：LSN 是 `db.log` 字节偏移，提交日志必须同步落盘，任何脏页、记录/索引文件头或 DDL 文件变更写盘前都必须保证对应日志已持久化；日志尾部不完整记录应在启动分析阶段截断。
- DML 日志保存记录前后镜像，索引日志保存完整索引文件名、二进制键和 RID；恢复完成后从基表重建全部索引，避免在半完成的 B+ 树分裂或合并上继续逻辑重放。
- DDL 固定为自动提交并通过全局 schema 互斥串行执行，不得放入显式事务；DROP 使用事务专属 tombstone 文件，提交后删除，未提交或崩溃时恢复原名和 `db.meta`。
- 当前恢复实现不写 CLR；运行时回滚必须先将撤销后的表页、索引页、文件头和元数据刷盘，再持久化 `ABORT`。恢复分析可将存在完整 `ABORT` 的事务视为已结束，禁止再次 UNDO，以免覆盖后续已提交事务。

## 实现优先级

除非用户明确指定其他目标，补功能时优先尊重现有架构和依赖顺序：

1. `storage` / `replacer`
2. `record`
3. `index`
4. `system`
5. `execution`
6. `transaction`
7. `recovery`
8. `parser` / `analyze` / `optimizer`

底层模块未完成时，不要只在上层执行器里绕过问题。比如扫描、插入、删除、索引查找等行为应尽量落在对应的 record、storage 或 index 抽象中实现。

## 测试与验证

实现功能或修 bug 后，应补对应测试，或至少运行相关已有测试并说明覆盖范围。常用验证方式包括：

- CMake 构建整个项目。
- 运行 `unit_test` 验证 LRU、DiskManager、BufferPoolManager、Record Manager 等底层行为。
- 启动 `rmdb` 服务端和 `rmdb_client` 客户端，使用手动 SQL 验证 DDL、DML、查询结果和错误路径。
- 检查输出格式时关注 `output.txt`，因为部分比赛测试会读取该文件。

新增文档或纯说明改动无需编译，但应检查 `git diff` 确认只改了预期文件。

Agent 不要自行执行 CMake 配置、构建或相关测试；仅在用户明确要求时运行。

## Server Failed 排查经验

- 服务端入口 `client_handler` 应把 SQL 解析、语义分析、计划生成和执行放在统一的异常边界内。解析阶段调用的 `std::stoll` 等函数也可能抛出异常，不能只捕获执行阶段错误，否则异常逃逸会直接导致连接线程或服务端失败。
- parser 的 `buffer_mutex` 和 `YY_BUFFER_STATE` 必须在成功、语法错误、RMDB 异常、事务中止及标准库异常等所有路径中各清理一次。优先使用统一、幂等的清理逻辑，避免分支中的漏解锁、重复 `yy_delete_buffer` 和死锁。
- 单语句隐式事务只有在请求完整成功后才能自动提交；任一阶段失败都应回滚。`TransactionAbortException` 已执行回滚时要记录状态，避免二次 abort。
- 错误响应写入 `data_send` 时必须按 `BUFFER_LENGTH` 截断并正确维护终止符和 `offset`；同时按评测约定向 `output.txt` 写入 `failure` 或 `abort`。
- 请求级 `Context` 优先使用作用域对象，连接结束时释放动态响应缓冲区。修复 Server Failed 时应同时检查异常生命周期、parser 资源、事务状态和 socket 响应，不能只在报错点增加一个 `catch`。
- parser 动作需要的通用错误与严格整数解析逻辑应放在 `errors.h` 等低依赖公共位置，避免为使用一个辅助函数引入 `common/common.h` 形成不必要的包含耦合；修改 `yacc.y` 后应保持生成的 parser 文件一致。
- 服务端应忽略 `SIGPIPE`，客户端超时或主动断开后的 socket 写失败只能结束当前连接，不能终止整个服务端进程。
- 顺序扫描筛选出的当前记录应在执行器内缓存，避免 `skip_unmatched()` 与 `Next()` 对同一记录重复读取、分配和复制；推进游标时必须同步清理缓存。

## 禁止事项

- 不要随意修改比赛 PDF、说明文档附件、第三方依赖、生成文件和无关 `.DS_Store`。
- 不要删除或重命名用户没有要求处理的文件。
- 不要使用破坏性 git 命令回滚工作区。
- 不要把临时调试输出、一次性实验代码或本地绝对路径留在源码中。
- 不要为了通过某个局部测试硬编码 SQL、表名、文件名或测试数据。

## AGENTS.md 自维护规则

当用户提出新的长期偏好、项目约束、调试经验、踩坑结论、测试流程或模块实现约定时，Agent 应主动判断这些内容是否具有复用价值。

如果具有复用价值，应在同一次任务中同步更新 `AGENTS.md`，让后续 Agent 能继承这些规则。更新时保持简洁，只记录稳定、可复用的项目约束，不把临时聊天内容、一次性决策或未验证猜测写入本文件。
